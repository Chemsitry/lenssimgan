1. The Problem: Simulations Don't Look Real

Your pipeline uses lenstronomy to forward-model gravitational lensing: given a lens mass distribution and a source galaxy, it ray-traces light and produces a synthetic 125×125 pixel image. These simulations are physically correct, but they're missing a crucial ingredient — the instrument.

Real JWST images (or VELA mock observations) have:

  • Correlated noise from the detector and background sky
  • PSF blurring from the optics and diffraction
  • Compression artifacts from the detector readout
  • Neighbor contamination from nearby galaxies in the field
  • Realistic flux calibration quirks

If you train a neural network classifier on clean simulations and then run it on real data, it will fail — not because the physics is wrong, but because the texture and noise structure are different. This is the domain gap problem.

The SimGAN's job is to bridge this gap: take a lenstronomy simulation and retouch it so it looks like it was observed by JWST, while keeping the lensing geometry intact.

Physics analogy: Think of it like a forward-model correction. You have a theoretical prediction (the lenstronomy image), and you want to add the instrument response function — but instead of specifying the IRF analytically, you learn it from data. The GAN is your empirical IRF estimator.

2. The GAN Idea — The Two-Player Game

A GAN involves two networks competing against each other. In your case, there's a specific variant called a SimGAN (Shrivastava et al. 2017), designed for exactly this simulation-refinement task.

The two players are:

Network Input Output Goal
Refiner R Simulated image sim Refined image ref Make ref look real enough to fool D
Discriminator D Any 125×125 image Probability ∈ [0, 1] Correctly label real=1, refined=0

Neither network is a generator in the classic GAN sense. The Refiner doesn't create images from noise — it edits existing simulations. This makes it much more constrained and stable to train, and it preserves the physics.

The game reaches a (theoretical) equilibrium when:

  • D cannot distinguish refined images from real ones (outputs 0.5 for everything)
  • R has learned to add just the right texture/noise to close the domain gap
Game theory framing: This is a minimax game. D wants to maximize its classification accuracy; R wants to minimize D's accuracy. At Nash equilibrium — if it exists — both players are stuck: R can't improve without D getting better, and D can't improve because R's output is indistinguishable from real data.

3. Network Architectures in Detail

3a. The Refiner (R)

The Refiner is a fully convolutional residual network. "Fully convolutional" means it contains no dense (fully-connected) layers — every operation slides a small filter across the image. This gives it a key property: it can only make local edits. A pixel in the output depends only on a neighborhood of pixels in the input, not the entire image at once.

Input (batch, 1, 125, 125) ← simulated image, 1 channel (grayscale) │ ▼ Conv2d(1 → 64, kernel=3×3, pad=1) ← 64 "feature detectors" slide across image ReLU ← nonlinearity: max(0, x) │ ▼ ×4 ┌─────────────────────────────────────────────────────┐ │ Conv2d(64 → 64, kernel=3×3, pad=1) │ │ BatchNorm2d │ │ ReLU │ │ Conv2d(64 → 64, kernel=3×3, pad=1) │ │ BatchNorm2d │ │ output = block(x) + x ← SKIP CONNECTION │ └─────────────────────────────────────────────────────┘ │ ▼ Conv2d(64 → 1, kernel=1×1) ← compress back to 1 channel Sigmoid ← squash to [0, 1] │ ▼ Output (batch, 1, 125, 125) ← refined image, same shape as input

What is a residual block / skip connection?

The key innovation is the skip connection: output = block(x) + x. Instead of the block learning to produce the refined image directly, it learns to produce the correction (the delta). If the block outputs zeros, the image passes through unchanged — this is an identity mapping.

Physics analogy: Think of it like perturbation theory. The block computes the first-order correction δ to the zeroth-order solution x. The skip connection enforces that the network starts near the identity and can only add small perturbations — exactly what you want for a refiner that shouldn't destroy the lensing geometry.

What does BatchNorm do?

Batch Normalization (BatchNorm) normalizes the activations across each mini-batch to have mean≈0 and variance≈1. This stabilizes training by preventing activations from exploding or vanishing. It also has a slight regularizing effect. It has two learnable parameters per channel: a scale γ and shift β, so it can undo the normalization if needed.

Padding=1 on 3×3 convolutions

Without padding, each convolution would shrink the image by 2 pixels on each side. With pad=1, a 1-pixel border of zeros is added before the convolution, so the output stays exactly 125×125. The spatial resolution is preserved throughout the Refiner.

The 1×1 exit convolution

The final Conv2d(64→1, kernel=1×1) is a weighted sum across the 64 feature channels at each pixel independently. It collapses the 64-channel feature map back to a single-channel image. Since kernel=1×1, there is no spatial mixing — it's a pure per-pixel linear combination of features.

3b. The Discriminator (D)

The Discriminator is a strided convolutional classifier. It takes a 125×125 image and outputs a single number in [0,1] — its estimate of the probability the image is real.

Input (batch, 1, 125, 125) │ ▼ Conv2d(1→32, kernel=4×4, stride=2, pad=1) → (batch, 32, 62, 62) LeakyReLU(0.2) Dropout(0.3) │ ▼ Conv2d(32→64, kernel=4×4, stride=2, pad=1) → (batch, 64, 31, 31) BatchNorm2d(64) LeakyReLU(0.2) Dropout(0.3) │ ▼ Conv2d(64→128, kernel=4×4, stride=2, pad=1) → (batch, 128, 15, 15) BatchNorm2d(128) LeakyReLU(0.2) Dropout(0.3) │ ▼ Conv2d(128→1, kernel=15×15) → (batch, 1, 1, 1) Sigmoid .view(-1) → (batch,) │ ▼ Output (batch,) ← one probability per image

Stride-2 convolutions vs. pooling

Each strided convolution (stride=2) halves the spatial resolution while doubling the channel count. This is a learned downsampling — the network decides which spatial information to keep, unlike max-pooling which just takes the largest value in each 2×2 block.

LeakyReLU instead of ReLU

A standard ReLU outputs 0 for any negative input, which means a neuron can permanently "die" — once its output goes negative, it never fires and its gradient is always zero. LeakyReLU instead outputs 0.2 × x for negative x, keeping a small gradient flowing. This matters especially in the Discriminator, where gradients need to reach the Refiner.

Dropout(0.3)

During each training step, 30% of activations are randomly zeroed out. This prevents the Discriminator from memorizing specific images and forces it to learn robust, generalizable features. It also acts as a regularizer to prevent it from becoming too strong and overwhelming the Refiner.

The 15×15 final convolution — global pooling

After three stride-2 convolutions, the 125×125 image has been compressed to 15×15. A Conv2d with a 15×15 kernel covers the entire remaining spatial extent at once. The result is a single number per image — equivalent to global pooling, but learned rather than fixed. This is how the Discriminator reduces a 2D image to a scalar probability.

4. The Loss Functions — Exactly

A "loss function" is simply a number that measures how badly a network is doing. The training algorithm tries to minimize this number by adjusting the network's parameters. Let's go through each loss term precisely.

4a. Binary Cross-Entropy (BCE) — The Shared Currency

Both the Discriminator and Refiner losses are built from Binary Cross-Entropy. For a single image with Discriminator output p and true label y ∈ {0, 1}:

BCE(p, y) = −[ y · log(p) + (1 − y) · log(1 − p) ]

When y = 1 (real image), this becomes −log(p) — large when p is small (confidently wrong), zero when p=1 (correct). When y = 0 (fake image), it becomes −log(1−p) — large when p is close to 1. The function is always ≥ 0 and equals 0 only when the prediction is perfect.

Physics analogy: BCE is the negative log-likelihood of a Bernoulli distribution. If you model each pixel classification as a coin flip with probability p of being "real," BCE is the information surprise (in nats) of seeing the true label. Minimizing BCE is equivalent to maximum likelihood estimation for a binary classifier.

4b. The Discriminator Loss

The Discriminator sees two batches per step: a batch of real images and a batch of refined images. Its loss is:

LD = ½ · BCE(D(xreal), 1) + ½ · BCE(D(R(xsim)), 0)

Breaking this down:

  • BCE(D(xreal), 1): For each real JWST/VELA image, D should output 1. This term penalizes D for calling real images fake.
  • BCE(D(R(xsim)), 0): For each refined simulation, D should output 0. This term penalizes D for calling refined images real.
  • Factor of ½: Equal weighting between the two error types. D is being trained to be a balanced classifier, not biased toward one class.

Note that when computing the Discriminator loss, the refined images R(x_sim) are computed with no gradient tracking on R (via torch.no_grad()). This is critical — the Discriminator update should only change D's parameters, not R's.

4c. The Refiner Loss

The Refiner loss has two terms pulled in opposite directions:

LR = Ladv + λ · Lpres

Term 1: The Adversarial Loss Ladv

Ladv = BCE(D(R(xsim)), 1)

This is the same BCE formula, but now with label=1 even though the image is refined (fake). The Refiner wants D to output 1 for its refined images — to fool D into thinking they're real. So R is minimizing the cost of D being wrong about its output.

Crucially, this time the gradient does flow back through D and into R. When Ladv is large (D confidently outputs 0 = "fake"), the gradient tells R: "your refinement looks unrealistic — change your weights to produce outputs that D scores higher." D's weights are frozen during this step.

Term 2: The Preservation Loss Lpres

This term prevents the Refiner from simply outputting the average JWST image. Without it, R would quickly learn to produce images that look real by ignoring the simulation entirely — destroying all the lensing physics.

The preservation loss penalizes R for changing the simulation too much. Your code offers two options:

Option A: MSE (Mean Squared Error)

LpresMSE = (1/N) Σi,j (R(xsim)ij − xsim,ij)2

This is just the average squared pixel difference between the refined image and the original simulation. Simple and differentiable everywhere. The problem: it treats all pixels equally and heavily penalizes large changes anywhere, even in regions where the noise structure needs to change a lot.

Option B: SSIM (Structural Similarity Index) — default

SSIM is more sophisticated. Rather than comparing pixel values directly, it compares local statistics: luminance, contrast, and structure. See Section 7 for the full breakdown. The preservation loss is:

LpresSSIM = 1 − SSIM(R(xsim), xsim)

Since SSIM ∈ [0, 1] (1 = identical images), this loss is 0 when the refined image is identical to the simulation, and approaches 1 when it's completely different. This is more tolerant of noise changes but sensitive to structural distortions.

The λ balancing parameter

The hyperparameter λ (default=10.0 for SSIM) controls the trade-off:

  • Too large λ: R barely modifies the simulation. Lpres dominates. The domain gap persists — refined images still look simulated.
  • Too small λ: R ignores the simulation structure. Ladv dominates. Refined images look real but the lensing geometry is destroyed.
  • λ=10 (SSIM default): Both terms are on a similar scale. Ladv ∈ [0, log(2)] ≈ [0, 0.69]; LpresSSIM ∈ [0, 1]. λ=10 gives preservation about 10× more weight per unit, favoring physics preservation.
Why λ≈200 for MSE? Simulated images are normalized to [0,1] and lens arcs are faint — typical pixel values are ≈0.05. So (Δpixel)² ≈ 0.0025 per pixel, and the mean over 125×125=15625 pixels gives LpresMSE ≈ 0.003 even for moderate changes. Meanwhile Ladv ≈ 0.5 — 100× larger! Without λ≈200, the MSE term would be irrelevant.

5. The Training Loop — Step by Step

Training proceeds epoch by epoch. Each epoch iterates through both datasets in synchronized mini-batches. Here is what happens in a single iteration: 

for each (real_batch, sim_batch) in zip(real_loader, sim_loader):

    ┌─────────────────────────────────────────────────────────┐
    │  STEP 1: Update Discriminator                           │
    │                                                         │
    │  refined = R(sim_batch)       # forward pass through R  │
    │  (no gradients recorded for R)                          │
    │                                                         │
    │  opt_D.zero_grad()            # clear old gradients     │
    │  L_D = 0.5*(BCE(D(real_batch), 1)                       │
    │           + BCE(D(refined), 0))                         │
    │  L_D.backward()              # compute ∂L_D/∂(D params) │
    │  opt_D.step()                # update D params only     │
    └─────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────┐
    │  STEP 2: Update Refiner                                 │
    │                                                         │
    │  refined = R(sim_batch)       # fresh forward pass      │
    │  (gradients ARE recorded for R and D)                   │
    │                                                         │
    │  opt_R.zero_grad()            # clear old gradients     │
    │  L_R = BCE(D(refined), 1) + λ * L_pres(refined, sim)   │
    │  L_R.backward()              # compute ∂L_R/∂(R params) │
    │  opt_R.step()                # update R params only     │
    └─────────────────────────────────────────────────────────┘

Why two separate forward passes?

In Step 1, torch.no_grad() means PyTorch doesn't build the computation graph for R. This is an optimization — we don't need gradients w.r.t. R, and not building the graph saves memory and computation. In Step 2, we do a fresh forward pass so that a full computation graph exists from R's weights through to the loss. If we tried to reuse Step 1's refined images in Step 2, there would be no graph connecting the loss to R's weights, and backward() would fail.

Why are the optimizers updated separately?

opt_D.step() only modifies D's parameters; opt_R.step() only modifies R's parameters. Even though the forward passes send gradients through both networks, each optimizer only applies the gradients for its own network. The parameters that get updated are exactly the ones passed to each optimizer at construction: 

opt_R = Adam(R.parameters(), lr=1e-4, betas=(0.5, 0.999))
opt_D = Adam(D.parameters(), lr=1e-5, betas=(0.5, 0.999))

Why is D's learning rate 10× lower than R's?

The Discriminator can learn very quickly relative to the Refiner — it's a simpler task (classify) than the Refiner's task (edit convincingly). If D learns too fast, it becomes near-perfect before R has a chance to improve, the adversarial gradient saturates (loss ≈ 0, gradient ≈ 0), and R stops learning. The lower learning rate keeps D from running away.

Adam optimizer — betas=(0.5, 0.999)

Adam maintains a running average of past gradients (β₁=0.5, short memory) and past squared gradients (β₂=0.999, long memory). It uses these to adapt the step size per parameter. The unusual β₁=0.5 (instead of the default 0.9) was recommended in the original DCGAN paper and makes the optimizer respond more quickly to gradient sign changes — useful in the adversarial setting where gradients can flip direction.

Checkpoint and visualization

Every 10 epochs (by default), the code:

  • Saves R and D weights to checkpoints/ as .pt files
  • Saves a loss plot showing L_D, L_adv, L_pres over iterations
  • Saves a sample grid showing three columns: simulated | refined | real — for visual inspection

6. What "Gradient" Means Here

Every parameter in R and D is a floating-point number. The entire training procedure is asking: "which direction should I nudge each number to reduce the loss?" The answer is the gradient — the partial derivative of the loss with respect to each parameter.

Consider a single weight w in the Refiner's first convolution layer. It affects the loss via a long chain:

w → (feature maps) → (refined image) → D(refined) → BCE → L_R

The chain rule computes:

∂LR/∂w = (∂LR/∂D(refined)) · (∂D(refined)/∂refined) · (∂refined/∂w)

PyTorch calls loss.backward() to compute all these partial derivatives automatically using reverse-mode automatic differentiation (backpropagation). It traverses the computation graph backward from the loss, accumulating gradients at each parameter using the chain rule.

Adam then updates: w ← w − lr · (adjusted gradient). The "adjusted" part is Adam's per-parameter learning rate scaling.

Physics analogy: Think of the loss landscape as a potential energy surface over a very high-dimensional space (one dimension per parameter — your Refiner has ~260k parameters). Training is gradient descent on this surface: at each step, you compute the local slope and take a step downhill. Adam is a sophisticated version that adapts the step size based on the local curvature history.

7. The SSIM Loss — Why Not Just MSE?

MSE treats every pixel difference identically. A 0.1 shift in a blank sky pixel costs the same as a 0.1 shift in the center of the Einstein ring. SSIM was designed to mimic human visual perception — we're sensitive to structure and contrast, not raw pixel values.

How SSIM is computed

For two image patches x and y of size 11×11 (weighted by a Gaussian window with σ=1.5):

Quantity Formula What it captures
μx, μy Gaussian-weighted mean of x and y Local luminance
σx², σy² Gaussian-weighted variance Local contrast
σxy Gaussian-weighted covariance Local structural correlation
SSIM(x, y) = [(2μxμy + C₁)(2σxy + C₂)] / [(μx² + μy² + C₁)(σx² + σy² + C₂)]

where C₁ = (0.01)² = 10⁻⁴ and C₂ = (0.03)² = 9×10⁻⁴ are small stabilization constants to avoid division by near-zero denominators. Your code computes this for every pixel location using a convolution with a Gaussian kernel, then takes the mean over all pixels:

LpresSSIM = 1 − (1/HW) Σi,j SSIMij(R(x), x)

Why Gaussian-weighted rather than flat-window?

Pixels near the center of an 11×11 patch contribute more than pixels at the edge. This reduces blocking artifacts — SSIM values computed with a flat window show discontinuities at patch boundaries. The Gaussian kernel (σ=1.5) effectively gives a smooth, overlapping local estimate.

Why does SSIM tolerate noise better than MSE?

Noise in an image is random and spatially uncorrelated. In any 11×11 patch, noise has near-zero covariance with a reference image (σxy ≈ 0), but it also appears in both σx² and σy². The SSIM formula factors these out — a noisy but structurally similar image still scores high SSIM. MSE instead penalizes noise directly because it squares every pixel difference.

For your use case: you want the Refiner to add realistic noise (to close the domain gap), but you don't want it to move the arc positions. SSIM allows noise changes while protecting structural similarity — exactly the right prior for this problem.

Physics analogy: SSIM separates a signal into three independent components — mean flux (luminance), flux variance (contrast), and spatial correlation structure — and checks each separately. It's similar to how you might compare two CMB maps: you'd compare power spectra and correlation functions rather than pixel-by-pixel residuals.

8. The Data Pipeline

The two datasets (real and simulated) are loaded and preprocessed before training begins.

8a. Simulated Images (lenstronomy)

Source: /global/cfs/projectdirs/deepsrch/jwst_sims/sims.15-.5/

  • Loaded from images.npy (all simulations) and lensed.npy (binary flag)
  • Filtered to only lensed images (lensed==1) when lens_only=True
  • Each image is independently normalized: img = (img − min) / (max − min) → [0, 1]
  • Reshaped to (N, 1, 125, 125) — the "1" is the channel dimension (grayscale)

8b. Real Images — Two Options

JWST GOODS-S Cutouts (--real-source jwst)

  • Source: GOODS-S F115W mosaic + JADES catalog of galaxy positions
  • 125×125 postage stamps extracted at each catalog position using WCS (world coordinate system) to convert RA/Dec → pixel coordinates
  • Up to 5000 cutouts by default, with a stride parameter to avoid crowded extractions
  • Per-image min-max normalization

VELA Cosmological Simulations (--real-source vela, default)

  • Source: VELA galaxy simulation suite, mock JWST observations in F115W
  • Central 125×125 crop from 800×800 FITS frames
  • Blank frames filtered out (mean pixel value < 10⁻⁵)
  • Per-image min-max normalization
  • Advantage: Cleaner than raw JWST (no cosmic rays, no neighbor contamination), but still has realistic noise texture and PSF

8c. DataLoader mechanics

Both datasets are wrapped in PyTorch TensorDataset → DataLoader: 

real_loader = DataLoader(TensorDataset(real_t),
                         batch_size=32, shuffle=True, drop_last=True)
sim_loader  = DataLoader(TensorDataset(sim_t),
                         batch_size=32, shuffle=True, drop_last=True)

The training loop zips both loaders: zip(real_loader, sim_loader). This means each iteration gets one batch of real and one batch of simulated images. They are shuffled independently — there's no pairing between individual real and simulated images. The number of iterations per epoch is determined by whichever dataset is smaller.

drop_last=True discards any final partial batch. This keeps batch sizes consistent, which matters for BatchNorm (which computes statistics over the batch — a batch size of 1 would give a degenerate estimate).

10. Physical Intuition for Each Piece

What the Refiner is actually learning

After training converges, the Refiner has learned a mapping from clean lenstronomy images to images with a realistic noise texture and PSF. Concretely, for each pixel in the simulation, the Refiner applies a learned local filter (via its residual blocks) that adds correlated noise, softens sharp edges to match the JWST PSF, and adjusts the background level.

Because of the residual structure, you can decompose the output as:

R(xsim) = xsim + Δ(xsim)

where Δ is the learned correction. If you subtract the two, you get the pure instrument response learned by the network. This correction is not a fixed PSF convolution — it's image-dependent, because the residual blocks are nonlinear and can apply different corrections to bright arcs vs. faint background.

What the Discriminator is actually learning

The Discriminator learns to detect features that are present in real JWST/VELA images but absent in lenstronomy simulations: the specific noise covariance structure of the detector, the wing profile of the JWST PSF on faint sources, the way sky background fluctuates between exposures. These are implicit — the Discriminator doesn't output an explanation, just a binary judgment.

After training, the Discriminator is typically discarded. Its value was in generating the adversarial gradient to guide the Refiner's training; it's not needed at inference time.

The equilibrium condition

Ideally, training converges to a point where:

  • D(xreal) ≈ 0.5 — D can't distinguish real from refined
  • D(R(xsim)) ≈ 0.5 — same
  • LD ≈ 0.5 × BCE(0.5, 1) + 0.5 × BCE(0.5, 0) = log(2) ≈ 0.693
  • Ladv ≈ BCE(0.5, 1) = log(2) ≈ 0.693

In practice, GANs rarely reach perfect equilibrium — the losses oscillate. What you watch for instead is that the sample grid (sim | refined | real) shows the refined images progressively gaining the noise texture and PSF profile of the real images, without losing the arc geometry.

Mode collapse: A failure mode where R learns to always output the same image (or a small set of images) that fool D. Signs: Ladv drops to near 0 very quickly, the refined column of the sample grid looks identical across rows. The preservation loss λ helps prevent this — if R collapses to a single output, Lpres explodes because all refined images are far from their corresponding simulations.

Why this approach for gravitational lensing

The domain gap in gravitational lens searches is severe because:

  1. Real lenses are extremely rare — ≲1 in 10,000 sources — so any systematic difference between sim and real distributions directly inflates false negatives
  2. The JWST PSF is complex and spatially variable across the focal plane
  3. The background contamination (neighboring galaxies, intracluster light) is not easily forward-modeled

SimGAN addresses (2) and (3) without requiring an explicit model of the instrument. The Discriminator implicitly learns whatever differentiates the two domains, and the Refiner is trained to close that gap — all from data, no hand-crafted instrument model required.
